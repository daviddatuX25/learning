What This System Is

Flexiqueue is a queue management system for government social welfare offices (MSWDO). It replaces paper tickets and manual shouting with a coordinated multi-surface system: kiosks, display TVs, staff panels, and mobile status pages — all communicating over WebSockets on a local LAN.

📌 Mission connection: Flexiqueue and SecureCAT-v2 share the same stack (Laravel 12 + Inertia v2 + Svelte 5). Every architectural pattern here — service layer, WebSocket strategy, offline resilience, role-based access — has a direct analog in SecureCAT-v2. The goal is to extract why each pattern was chosen.

📐 Architecture Layers (Laravel Standard + Extensions)

1. Routes → Controllers

The route file (routes/web.php, 595 lines) is heavily commented with cross-references to specification documents. Routes are grouped by permission middleware, not by surface:

  • permission:admin.manage — Program/station/token CRUD, analytics, print/TTS settings
  • permission:platform.manage — Super-admin platform settings (TTS budgets, ElevenLabs)
  • permission:admin.shared — Users, sites, logs, RBAC (shared between admin and super_admin)
  • permission:staff.operations — Session lifecycle (bind, call, serve, transfer, complete, cancel)
  • permission:profile.self — User profile, PIN/QR override, password
  • permission:dashboard.view — Dashboard stats
  • permission:auth.supervisor_tools — Temporary PIN/QR generation
  • No auth — Public routes (kiosk, display, status check, triage)
🧠 For SecureCAT-v2: Flexiqueue's permission middleware pattern (Spatie groups per route group, not per route) is cleaner than individual middleware on each route. Consider grouping your AI and ML endpoints behind permission:ai.manage etc.

2. Controller → Service Layer

Controllers are thin. DisplayController.php is 33KB but most is Inertia page rendering. Business logic lives in 50+ service classes under app/Services/:

🧠 For SecureCAT-v2: Notice services depend on other services via constructor injection (e.g., SessionService takes FlowEngine, StationSelectionService, PinService, etc.). This composes domain logic from small, testable units. Your OMR service and AI Triage service should follow the same pattern — inject the AI SDK client service, don't call it from the controller.

3. The FlowEngine (State Machine Pattern)

The FlowEngine is a pure routing algorithm — 74 lines with no database writes. Given a Session, it determines the next station/process:

// Simplified logic:
function calculateNextStation(Session $session) {
    // 1. If session has override_steps (custom one-off path), use those
    // 2. Otherwise, query TrackStep where step_order = current + 1
    // 3. If that TrackStep has a process_id, return it (caller picks station via StationSelectionService)
    // 4. Return null = flow complete
}
  • Normal flow: A token is bound to a Track (e.g., "Registration"). The track has ordered TrackSteps (Step 1: Registrar Window, Step 2: Cashier, Step 3: Library). The FlowEngine walks through them one by one.
  • Override flow: A supervisor can assign override_steps (an array of station IDs) to handle one-off routing ("go to Window 5 directly").
  • Process → Station selection: The PROCESS-STATION-REFACTOR decoupled processes (work types) from stations (physical desks). Now a TrackStep points to a Process, and StationSelectionService picks the best station for that process.
🧠 For SecureCAT-v2: If you implement a "Service Triage" state machine (student → triage → AI assessment → OMR → completion), a FlowEngine-like class keeps routing logic isolated and testable. The override_steps pattern is useful for manual override when AI triage suggests a non-standard path.

🗄️ Data Model (Core Entities)

┌──────────┐ ┌──────────┐ ┌──────────────┐ │ Program │───>│ Track │───>│ TrackStep │ │ (active) │ │(service) │ │(ordered step) │ └──────────┘ └──────────┘ └──────┬───────┘ │ │ │ ┌─────▼──────┐ │ │ Process │ │ │ (work type) │ │ └──────┬──────┘ │ │ │ ┌──────▼──────┐ │ │ Station │ │ │ (desk/window)│ │ └──────┬──────┘ │ │ ┌──────▼──────┐ ┌──────▼──────┐ │ Token │ │ Session │ │ (unique QR) │<────────────────>│(queue_sessns)│ └──────────────┘ bound to └──────┬──────┘ │ ┌──────▼──────┐ │TransactionLog│ │ (append-only)│ └─────────────┘

Key schema decisions worth studying in the migrations:

  1. Token → Session separation: A token is a static identifier (printed ticket with QR). A session is a dynamic journey (token + track + stations visited). One token can be used in multiple sessions if it re-queues. This separation is smart — it decouples identity from state.
  2. Append-only TransactionLog: Every queue action (bind, call, serve, transfer, complete, cancel) is inserted, never updated or deleted. This gives COA-compliant audit trails. In SQLite, INSERT is extremely cheap and ACID-safe.
  3. Program settings as JSON column: Program has a settings JSON column rather than 20 separate columns. This allows flexible per-program configuration (priority lane ratios, TTS settings, display preferences) without schema migrations.
  4. Station ↔ Process M:M: The pivot table station_process allows any station to serve any process. This decouples physical desk layout from workflow definition.

⚡ Realtime (WebSocket) Architecture

Flexiqueue uses Laravel Reverb on the edge (local LAN) and Pusher on the central (cloud) instance. Both implement the same broadcasting API — the .env file switches between them.

Key events fired:

  • NowServing — Triggered when staff calls a token. Display boards pick this up and update their UI + TTS.
  • QueueLengthUpdated — For dashboard widgets and mobile status.
  • ClientArrived — When a kiosk user prints a ticket.
  • StatusUpdate — Generic status change broadcast.
  • StationActivity — Station-level activity log.
🧠 For SecureCAT-v2: Reverb vs Pusher dual-config means you can develop locally with Pusher (no extra daemon) and deploy with Reverb (no external dependency). This is exactly the pattern you want for your AI Companion's realtime chat — use Pusher during dev, Reverb in production so you control the data.

🔐 RBAC & Authorization

Built on Spatie Laravel Permission with a custom sync layer (SpatieRbacSyncService). Route comments reference a SUPER-ADMIN-VS-ADMIN-SPEC spec:

  • Super Admin (platform.manage) — Platform-level settings, TTS budgets, integrations.
  • Admin (admin.manage) — Program/station/token CRUD, analytics, print settings.
  • Shared (admin.shared) — Users, sites, logs (both Super Admin and Admin).
  • Staff (staff.operations) — Session operations (call, serve, transfer).
  • Supervisor (auth.supervisor_tools) — Temporary PIN/QR generation.

There's also a device-level auth via auth.edge_device middleware (API tokens for edge devices, not user sessions).

🌍 Edge vs Central (Offline-First)

Flexiqueue has two deployment modes:

Edge (Orange Pi) Central (cloud)
DatabaseSQLite (zero-config, low-memory)MariaDB
WebSocketLaravel Reverb (local)Pusher (cloud)
Internet needed?❌ No — 100% offline✅ Yes
SyncPushes transaction logs → centralReceives + replays logs
DetectionEdgeModeService runtime detectionN/A

The edge sync is a transaction-log-based push: the edge device inserts into transaction_logs, and a background job POSTs unsynced logs to the central endpoint (POST /api/edge/sync). The central server replays them to reconstruct state. This is conceptually simpler than real-time replication: no conflict resolution needed because each site's tokens are scoped by site ID.

🧠 For SecureCAT-v2: If you have an offline component (e.g., OMR scanning on a Raspberry Pi), consider transaction-log push rather than full DB replication. It's simpler, COA-compliant, and avoids conflict resolution. Just scope records by device ID.

🔍 Key Architectural Patterns Summary

Pattern Flexiqueue Implementation Why It Matters
Service Layer Composition Services inject other services; controllers inject services Unit-testable domain logic, thin controllers
State Machine FlowEngine: pure function session → next station Routing logic centralized, no if/else spaghetti in controllers
Process/Station Separation M:M pivot, selection service picks best station Decouples workflow from physical layout
Append-Only Audit TransactionLog: INSERT-only, never UPDATE/DELETE Tamper-proof logging, COA compliance
Dual WebSocket Driver .env switch: Reverb (local) / Pusher (cloud) Dev with cloud, deploy without dependency
JSON Settings Columns Program.settings as JSON instead of 20 columns No migrations for new config knobs
Permission Route Grouping Routes grouped by permission, not by surface One middleware per group, Lazy loading, not per-route
Transaction-Log Sync Edge pushes logs; central replays them Simple offline sync without conflict resolution
