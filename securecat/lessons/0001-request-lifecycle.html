Lesson 1: The Laravel Request-Response Lifecycle

Tracing a real SecureCAT page from URL to browser

Mission: Understand how a URL like /admin/exam-scheduling becomes a fully rendered HTML page in the browser. This is the single most important mental model for Laravel development.

Why this matters

Every framework has a "pipeline" — a fixed sequence of steps every request goes through. When you know the pipeline, you stop guessing. You know exactly where to add a feature, where to fix a bug, and why errors happen at certain stages.

The route we're tracing

From routes/web.php (line ~200):

// Route registered inside middleware group 'auth' and 'role:...'
Route::middleware('role:super_admin,registrar_administrator,test_administrator,proctor')
    ->prefix('admin')
    ->name('admin.')
    ->group(function () {
        Route::get('exam-scheduling', [ExamSessionController::class, 'index'])
            ->name('exam-scheduling.index');
    });

The full pipeline (9 stages)

1
Entry — public/index.php

Every request to Laravel arrives here. Apache/Nginx rewrites all URLs to this file. It loads Composer's autoloader, then bootstraps the Laravel application instance and kernel.

// public/index.php (simplified)
$app = require __DIR__.'/../bootstrap/app.php';
$kernel = $app->make(Kernel::class);
$response = $kernel->handle(
    $request = Request::capture()
)->send();
$kernel->terminate($request, $response);
Junior note: You'll almost never edit this file. It's the "engine starter" — you just need to know it exists and that $kernel->handle() is where the pipeline begins.
2
Kernel — app/Http/Kernel.php

The HTTP Kernel defines middleware stacks. Every request passes through:

  • Global middleware — runs on every request (trusted proxies, maintenance mode check, etc.)
  • Web middleware group — session handling, CSRF protection, cookie encryption, HandleInertiaRequests
  • Route-specific middleware — whatever you attach to the route itself

For our route, the effective middleware stack (in order) is:

web group →
  auth →                    checks user is logged in
  role:super_admin,... →    checks user has one of these roles
  verified: true            (implicit — email verification if enabled)
Gotcha: The role middleware is registered as a named middleware in $routeMiddleware. In SecureCAT it's app/Http/Middleware/EnsureUserHasRole.php — just 29 lines.
3
Middleware: EnsureUserHasRole.php

Let's look at the actual code:

class EnsureUserHasRole
{
    public function handle(Request $request, Closure $next, string ...$roles): Response
    {
        if (! $request->user()) {
            return redirect()->route('login');
        }

        if (! $request->user()->hasAnyRole($roles)) {
            abort(403, 'Unauthorized. Required role: '.implode(' or ', $roles));
        }

        return $next($request);
    }
}

Two checks, 15 lines of logic. If either check fails, the request never reaches the controller — you get a redirect or 403 page instead.

Key insight: Middleware is a stack (or "onion"). Each layer can either pass the request inward ($next($request)) or short-circuit it (redirect/abort). The response travels back out through the same layers in reverse order.
4
Routing — routes/web.php

Once middleware passes, the Router matches the URL path /admin/exam-scheduling against all registered routes. It finds the match and calls the controller method.

Route::get('exam-scheduling', [ExamSessionController::class, 'index'])
    ->name('admin.exam-scheduling.index');

The route name (admin.exam-scheduling.index) lets you generate URLs in templates and controllers without hardcoding paths — route('admin.exam-scheduling.index') gives you /admin/exam-scheduling.

5
Controller — ExamSessionController.php

The index() method runs. Here's the simplified flow:

public function index(Request $request): Response
{
    // 5a. Authorize via Policy
    $this->authorize('viewAny', ExamSession::class);

    // 5b. Determine if this is a proctor view
    $isProctorView = $user->hasAnyRole(['proctor'])
        && ! $user->hasAnyRole(['super_admin', 'registrar_administrator']);

    // 5c. Query exam sessions with eager loading
    $query = ExamSession::query()
        ->with(['room:id,name,building,capacity', 'proctors:id,name'])
        ->orderBy('date')->orderBy('start_time');

    // 5d. Apply filters from query string
    if ($request->filled('status')) {
        $query->where('status', $request->get('status'));
    }
    // ... more filters ...

    $sessions = $query->paginate(15)->withQueryString();

    // 5e. Build payload & render via Inertia
    return Inertia::render('Admin/TestScheduling/Index', [
        'sessions' => $sessions,
        'filters' => $request->only([...]),
        // ... more data ...
    ]);
}
Junior note: Notice the controller doesn't build HTML directly. It just gathers data and passes it to Inertia. This is the "thin controller, fat service" pattern — SecureCAT moves complex logic into service classes.
6
Policy — app/Policies/ExamSessionPolicy.php

The $this->authorize('viewAny') call in step 5 checks the ExamSessionPolicy to see if the user is allowed to list exam sessions. If the policy returns false, Laravel throws a 403 exception before any data is queried.

class ExamSessionPolicy
{
    public function viewAny(User $user): bool
    {
        return $user->hasAnyRole([
            'super_admin',
            'registrar_administrator',
            'test_administrator',
            'proctor',
        ]);
    }
}

Two layers of authorization here: the middleware checks the role at the route level (gate), then the policy checks at the resource level (finer-grained). Redundant? Sometimes. But defense-in-depth is intentional — the policy also applies when calling from other contexts (like API or queued jobs).

7
Inertia Middleware — HandleInertiaRequests.php

Before Inertia sends the response, this middleware injects shared props — data available on every page. In SecureCAT, that includes:

public function share(Request $request): array
{
    return array_merge(parent::share($request), [
        'auth' => ['user' => $authUser],
        'flash' => [
            'success' => $request->session()->get('success'),
            'error'   => $request->session()->get('error'),
        ],
        'csrf_token' => $request->session()->token(),
        'notifications' => $user?->notifications()->latest()->limit(20)->get(),
        'activeAcademicYear' => $this->resolveActiveAcademicYear(),
        // ... more shared state ...
    ]);
}

These props are available in every Svelte page via $page.props. That's why AuthenticatedLayout.svelte can access $page.props.auth.user without it being passed in each controller.

Watch out: HandleInertiaRequests runs on every authenticated request. Avoid putting expensive DB queries here — they run on every page load, not just the one that needs them. SecureCAT loads notifications here, which is fine because it's limited to 20.
8
Inertia rendering — Server to Client

Inertia receives the controller's data and shared props, then:

  1. First visit: Renders a full Blade view (resources/views/app.blade.php) that loads the Svelte app. It bootstraps the page data as a JSON <div id="app" data-page="..."> attribute.
  2. Subsequent visits (SPA): Sends a JSON response with X-Inertia header. The client-side router swaps the component without a full page reload.
// This is what Inertia::render() does internally:
{
  "component": "Admin/TestScheduling/Index",
  "props": {
    "auth": { "user": { "id": 1, "name": "admin", ... } },
    "sessions": { "data": [...], "meta": { ... } },
    "filters": { ... },
    ...
  },
  "url": "/admin/exam-scheduling",
  "version": "abc123"
}
9
Svelte page — resources/js/Pages/Admin/TestScheduling/Index.svelte

The Svelte component receives sessions, filters, statuses, etc. as props from Inertia. It renders the exam scheduling table inside AuthenticatedLayout.svelte, which provides the sidebar, header, notifications dropdown, and theme toggle.

<!-- Simplified structure of the Svelte page -->
<script>
  let { sessions, filters, statuses, breadcrumbParent } = $props();
  let search = $state(filters.search ?? '');
  let statusFilter = $state(filters.status ?? '');

  function applyFilters() {
    router.get('/admin/exam-scheduling', {
      search, status: statusFilter
    }, { preserveState: true });
  }
</script>

<AuthenticatedLayout breadcrumbs={breadcrumbParent}>
  <div class="p-6">
    <h1>Exam Scheduling</h1>
    <input bind:value={search} placeholder="Search rooms..." />
    {#each sessions.data as session (session.id)}
      <!-- render each session row -->
    {/each}
  </div>
</AuthenticatedLayout>
Svelte 5 note: $props() is a rune — Svelte 5's new reactivity primitive. It replaces the old export let syntax. $state() creates reactive local variables. When they change, the DOM updates automatically.

Where technical debt hides in this flow

Stage What a junior might miss
Middleware role middleware runs after auth. If auth passes but role fails, the 403 page loads — but session data was already read. Not a perf issue here, but ordering matters when you add middleware that queries the DB.
Controller $this->authorize('viewAny') at line 51 — if you forget this, any role-based user could hit this page even if the middleware lets them through.
Eloquent ->with(['room:id,...', 'proctors:id,...']) — this eager-loads to prevent N+1 queries. Without it, every session in the loop would query the room separately. N+1 is the #1 performance bug in Laravel.
Shared Props HandleInertiaRequests queries notifications and academic year on every page. If the project grows, these should be lazy-loaded or cached.
Controller size ExamSessionController is 1,012 lines — too big. The scheduleAssistantPayload() private method (lines 122-204) really belongs in a Service class.
No tests Zero files in tests/. This means there's no automated safety net — you can't refactor with confidence.
